    Continue to select squares, but make sure to only click those that have something underneath them (clicking on floating spaces, or those at the top will display an error). Once a player has connected four of their tokens in a row, they have won the game!

The Basics

What is Connect Four ?
    Connect Four (also known as Four Up, Plot Four, Find Four, Captain's Mistress, Four in a Row, Drop Four, and Gravitrips in the Soviet Union) is a two-player board game. Players choose a color and then take turns dropping their colored tokens into a seven-column, six-row vertical grid.
The Goal ?
    A player's goal is to deposit their pieces in such a way that, eventually, they occupy four adjacent spaces - forming a connected line. This line can be hoirzontal, vertical, or even diagonal. Once a player has four discs in a row, they've won the game!
The Challenge ?
    A player needs to think vertically : pieces can only be inserted at the top of the grid. Once inserted, they fall. The results of player moves can produce a visually-disorganized playing field, with bright reds, blues, and yellows scattered throughout the grid. To overcome these obstacles, one must
  • continually scan the field, searching for potentially-useful patterns for both one's self, and one's opponent, and
  • use anticipatory strategy to set up (and obstruct) connected lines.
Developer Stories

Conceptual Challenge # 1

Faking Gravity ...
    On its face, this app could have been a lot like Checkers, Go, or Minesweeper: each game has a horizontal board that allows players to place pieces at-will. However, Connect Four's grid-board stands up vertically in reality. Gravity's essential role, then, produces two key problems:
  • "Replicate a 'Falling' action"

    • While using the app, if a player were to click, say, a space on the top row one might expect that specific grid space to be taken. However, what ultimately needs to happen is for the player's piece to fall.

    The solution: for a MVP app, I chose to force users to do most of the work - they have to find the final landing place of their token on their own (rather than have the app do it for them).

  • "It needs a floor"

    • If a player inserts a piece into one of the seven columns, it falls until it collides with something. In the physical game, there is a slider at the bottom of the grid, which blocks any piece from falling out onto the table. Once a piece has been deposited on the first row, more pieces can be stacked on top of it.

    The solution: replicate that slider with an eighth hidden row. All the spaces in that row are pre-set to "taken," which then allows them to be checked just like any other.

Conceptual Challenge # 2

Confirming Win Condition ...

    In theory, both human and app should be able to look at currently-occupied spaces on the board, and then track the vectors outwards in any of the nine possible directions. However this logic is far more complicated than is needed in order to make the app work. In order to make this app reach MVP status, I used a work around ... vertically.

    The solution: I chose to use an array of "winning number" solutions. Given that each space on the vertical board is uniquely-numbered, if one took the time, one could work out every set of four grid spaces that creates a straight line.

    const winningArrays = [
    _____[0, 1, 2, 3], __// Four horizontal
    _____[0, 7, 14, 21], // Four vertical
    _____[0, 8, 16, 24], // Four diagonal
    _____... etc. ]

    With this, the app can then confirm whether all spaces refered to in a set or owned by one player; if 'yes', then that player has won! Thankfully, this list already existed online.

Ex: if the neighbor is 6 spaces away, it's diagonally-adjacent, and so-to would any valid neighbors.

Development Challenge

Making the app 'see' ...

    Problem: the app needs to be able to read the grid as humans would, and confirm whether there are any four-segment lines after every move. Solution: Loop through the grid (to be up-front, I did a lot of googling to figure this out) ...

    -
    First, the board/grid is made by wrapping 42 sibbling 'divs' inside of a parent 'div':
    < div id="grid">
    ----< div>
    ----< div>
    ----< div>
    ----. . . etc.
    Second, these 'divs' are stored in a JS variable:
    const squares = document.querySelectorAll('#grid > div')
    -
    Third, the app 'looks' at them by using a loop:
    for (let y = 0; y < winningArrays.length; y++) {
    const square1 = squares[winningArrays[y][0]]
    const square2 = squares[winningArrays[y][1]]
    const square3 = squares[winningArrays[y][2]]
    const square4 = squares[winningArrays[y][3]]
    -
    I implemented the solution, and it worked beautifully.
    -
    Comprehension Problem: when I went back to do 'final checks' on my code, this loop completely confused me - "How exactly is it *seeing* these squares??" How are the 'divs' differentiated from one another without individual IDs etc?
    • It's obviously storing something in the square variables,
    • It's also obviously checking all four elements in each of winningArray's nested combinations,
    • ... But what is "squares" doing?
    -
    I resorted to using my bootcamp's "Peter Jang Notation" - where you spell out each thing that happens: "use 'y' to go through each sub-set of winningArrays, and then grab the 0th element in that sub-set; with that number in-hand (say, 39), get ?something? with a property of 39 in the DOM?" I'd obviously lost track of what was being stored in the 'squares' variable by this point, and having just done significant DOM manipulation studying, I was also getting side-tracked by 'attributes' and 'properties' concepts. I just couldn't figure out how the DOM elements were being stored in that variable.
    -
    Thank goodness there's Google. And Stack Overflow.
    -
    Clarity: After some more reading, it became clear that the loop wasn't checking DOM properties; it was using winningArrays' number to store the equivilent DOM element in the 'squares' "array." I'm still learning about the differences between a true array and HTML collections, but it hadn't occured to me that they could be searched by index place (if this is wrong, please tell me - I really, really want to make sure I've got this right! 😁)
    -
    This epiphany has greatly increased my ability to manipulate the DOM and navigate it - using the in-browser console is starting to feel like my local terminal! Learning is fun.
